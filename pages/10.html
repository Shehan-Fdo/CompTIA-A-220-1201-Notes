DNS (Domain Name System)

CompTIA A+ 220-1201 - Core 1 | Lesson 1.2

Introduction to DNS

  • DNS (Domain Name System) is a critical network service that translates fully qualified domain names (FQDNs) (e.g., www.professormesser.com) into IP addresses.
  • Eliminates the need to memorize IP addresses by allowing users to access resources using human-readable names.
  • Functions as a distributed, hierarchical database scattered across global servers.
  • Supports additional features beyond name resolution (e.g., email routing, security records).

DNS Hierarchy and Structure

  • Root Servers: 13 logical clusters (physically >1,000 servers) that direct queries to top-level domains (TLDs).
  • Top-Level Domains (TLDs):
    • Generic TLDs (gTLDs): .com, .org, .net.
    • Country-Code TLDs (ccTLDs): .us (US), .ca (Canada), .uk (UK). ~275 country codes exist.
  • Hierarchical Structure:
    • Example: katie.east.professormesser.com breaks down as:
      • com (TLD) → professormesser (domain) → east (subdomain) → katie (host).

DNS Tools

  • dig (Domain Information Groper):
    • Linux/Mac/Windows (with installation) command-line tool.
    • Queries DNS servers for records (e.g., dig www.professormesser.com).
    • Returns:
      • Query details (e.g., QUESTION section).
      • Answers (e.g., IP addresses, TTL).
  • nslookup:
    • Windows-native tool (also available on Linux/Mac).
    • Functionality similar to dig (e.g., nslookup www.professormesser.com).

DNS Resource Records (RR)

  • Stores data in resource records (RR), which define:
    • IP addresses (A/AAAA).
    • Mail servers (MX).
    • Aliases (CNAME).
    • Text-based info (TXT).
    • Authorization (SOA).
  • Critical Notes:
    • Misconfigurations can make devices unreachable.
    • Always backup DNS configurations before changes.

Key DNS Record Types

  • A Record:
    • Maps a hostname to an IPv4 address.
    • Example: www.professormesser.com → 162.159.246.164.
    • TTL (Time to Live): Duration (e.g., 15 mins) a record is cached locally.
  • AAAA Record (Quad A):
    • Maps a hostname to an IPv6 address.
    • Process identical to A records but for IPv6.
  • CNAME Record (Canonical Name):
    • Creates an alias for a hostname.
    • Example: ftp.example.com → mail.example.com (both point to the same IP).
    • Simplifies management: Changing the A record of the target host updates all CNAMEs.
  • MX Record (Mail Exchanger):
    • Specifies mail servers for a domain (critical for email delivery).
    • Example: IN MX mail.example.com.
    • Mail servers query MX records to route emails (e.g., james@professormesser.com → mail.professormesser.com).
  • TXT Record:
    • Stores human-readable text (e.g., verification keys, policies).
    • Common uses:
      • Verification: Proving domain ownership (e.g., for email services).
      • SPF/DKIM/DMARC: Email security (see below).
    • View with: dig professormesser.com TXT or nslookup -type=txt google.com.

Email Security Records

  • SPF Record (Sender Policy Framework):
    • TXT record listing authorized email servers for a domain.
    • Prevents spoofing by verifying the sending server’s IP.
    • Example: v=spf1 include:_spf.google.com ~all.
  • DKIM Record (DomainKeys Identified Mail):
    • Adds a digital signature to emails using a public/private key pair.
    • Public key is published in a TXT record; private key signs outgoing emails.
    • Recipient servers verify signatures using the public key.
  • DMARC Record (Domain-based Message Authentication, Reporting & Conformance):
    • TXT record defining policies for emails that fail SPF/DKIM.
    • Actions:
      • p=none: No action (monitor only).
      • p=quarantine: Send to spam.
      • p=reject: Block delivery.
    • Example: v=DMARC1; p=quarantine; rua=mailto:reports@professormesser.com.

DNS Management

  • Configuration Methods:
    • Text-based files: Direct editing (e.g., SOA, A, MX records).
    • Web-based frontends: User-friendly interfaces (e.g., cPanel, Cloudflare).
  • Best Practices:
    • Test changes in a non-production environment.
    • Use short TTLs before major changes (e.g., 5 mins) to allow quick propagation.
    • Monitor with tools like dig/nslookup post-update.
